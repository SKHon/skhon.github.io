关于一次502排查经历

发表于 2020-08-27 分类于 后端
本文字数： 0 阅读时长 ≈ 0:01

背景

业务方半夜滴我，说他们服务7层监控有一些502，没有头绪，帮我们看看。当时的我心里真的是翻江倒海的，心里一万只……但是业务方用的node框架是我负责的，那就帮忙看看吧

502会是什么问题

502：Bad Gateway，错误网关或者无效网关。示意图如下：

可以看到，就是上游服务无法响应了。

什么原因会导致502

  • 服务器超载——如果服务器耗尽内存，服务器可能会崩溃，原因是站点上有大量访问者或DDOS攻击。
  • 防火墙阻止请求——防火墙可能会阻止边缘服务器和上游服务器之间的通信。某些DDoS保护系统可能会过度反应并阻止来自内容传送系统的请求。
  • 编程错误——有时网站代码中的一个错误可能意味着请求不能得到正确的响应，从而导致出现这个错误。网络错误——DNS问题、路由问题和ISP相关的问题也可以导致错误网关。
  • 服务器软件超时——当Web服务器需要更多时间来完成并且缓存工具达到其超时值时，也会发生此错误。慢查询也会导致此问题。

回到业务方的问题

最后排查的时候，查了一下hobber日志，是axios报错，没有catch到，导致进程crash了，然后机器会试图再把这个服务重启。然后，该业务方服务没有接入平滑重启，导致服务重启的过程中，一些请求过来，就会无法响应，然后返回了502。那你是不是有个疑问，为什么axios的异常没有catch到，是因为try 中如果是异步中出现了异常，是catch不到的，如果想catch到这类异常，可以用一下这个hprocess.on(“uncaughtException”, (error) => {})

总结

其实一些5xx之列的问题排查，你得积累经验，如果你碰到了这类问题，你就去努力去解决，这次依赖别人，你要相信，你还会碰到这个问题的。我清楚记得我就有这种经历，当时有一个iframe嵌了一个koa应用，然后拿不到cookie，第一次我发现这个问题的时候，我觉得，这应该是业务方的问题，和我们框架没啥关系，就没有研究这个问题，后来业务方找过来，线上故障，请求帮忙解决。然后还得帮其解决，最后发现是sameSite策略的问题。我真的是又当爹又当妈，不过这个过程我积累了很多经验，也算对我的补偿吧！
